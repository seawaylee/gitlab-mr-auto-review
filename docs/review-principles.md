# Code Review 原则（当前版本）

这份文档描述当前自动化 CR 的默认原则，分为两部分：

1. 评审时希望遵循的判断原则
2. 代码里已经真实落地的行为边界

后续如果你要加入特殊要求，建议优先标注它属于哪一类：

- 原则升级：改变“什么情况下该拦、该提醒、该放行”
- 实现升级：改变 prompt、输出字段、规则校验、评论格式

## 1. 评审目标

当前自动化 CR 的目标不是替代人工审核，而是做一层稳定的风险筛查：

- 先快速说明这次 MR 在做什么
- 再给出明确的审查结论和风险等级
- 最后把真正值得人工关注的问题提出来

默认优先级是：

1. 正确性和回归风险
2. 兼容性和数据安全
3. 可维护性和测试完备性
4. 风格和表达一致性

也就是说，默认不鼓励把时间花在低价值的样式挑刺上，而是优先抓会影响合并质量的问题。

## 2. 信息来源与边界

当前评审主要基于以下输入：

- MR 标题
- MR 描述
- 源分支和目标分支
- 作者
- MR 链接
- diff 内容

当前默认边界也要说清楚：

- 不是基于完整仓库语义分析
- 默认不会主动运行测试、构建或启动服务
- 默认不知道业务需求文档、上线计划和历史事故上下文
- 如果 diff 之外缺少证据，不应凭空下结论

因此默认原则是：有证据就指出，没有证据就保守表达。证据不足时，优先给 `comment`，而不是武断给 `request_changes`。

## 2.1 有限扩展阅读

默认不应该只盯着 diff 本身看。

如果条件允许，每次 CR 都应在变更文件基础上做有限扩展阅读，至少向关联代码扩 2 到 3 层，但范围要有边界，避免演变成全仓库漫游。

默认优先扩展阅读这些位置：

- 变更文件自身的完整实现
- 直接调用的上下游依赖
- 关键入参、返回值、共享数据结构所在的类
- 与本次变更直接绑定的测试文件

扩展阅读时，默认重点排查：

- 上下游参数传递是否存在空值、缺省值、类型假设不成立的风险
- 返回值契约是否被过度乐观假设，例如默认非空、默认有值、默认不重复
- 分页、过滤、裁剪后只处理子集数据时，行为是否仍符合预期
- 中间层对象、Map、List、Optional、上下文字段在多层访问时是否容易出现空指针或兼容性问题
- 当前改动是否依赖外部组件的隐含行为，但代码里没有防御或说明

这里的关键是“有限”：

- 目标是补足关键上下文，不是穷举所有调用链
- 优先读最可能影响本次结论的 2 到 3 层代码
- 如果扩展上下文拿不到，就明确承认边界，不假装已经看全

## 3. 默认判断原则

### 3.1 风险优先，而不是面面俱到

默认先看高价值风险：

- 逻辑是否可能写错
- 是否容易引入回归
- 是否破坏旧接口、旧字段、旧行为
- 是否影响数据正确性、一致性、幂等性
- 是否缺少关键异常处理和边界保护

### 3.2 只提能站得住的结论

默认要求问题描述具备最少的可辩护性：

- 最好能从 diff 直接看出问题
- 最好能指出受影响文件、分支逻辑或场景
- 尽量避免“可能有问题但说不清哪里”的空泛表述

如果只能提出风险假设，也应该明确写成“建议关注”或“建议补充验证”，而不是伪装成确定性 bug。

### 3.3 关注变更带来的系统性后果

默认不仅看单行代码对不对，还看它会不会带来系统性风险，例如：

- 接口契约变化但没有兼容处理
- 状态流转变化但没有回归验证
- 错误码、返回体、日志字段、事件字段发生变化
- 配置、依赖、鉴权、缓存、并发控制被修改
- 缺少与高风险改动匹配的测试或兜底措施

还要补一条明确要求：

- 评估本次改动，对“非本次修改意图”的业务是否产生副作用

这条的意思不是发散到整个系统，而是要主动问一句：

- 这次改动本来想解决 A，会不会顺手影响到并不打算改的 B、C、D

默认优先关注这几类“非本次修改意图”的影响：

- 原本稳定的上下游调用方是否被被动改变了语义
- 通用组件、基础组件、公共方法被改动后，其他接入方是否可能一起受影响
- 参数默认值、空值处理、异常处理、返回结构调整后，是否会波及旧链路
- 分页、过滤、缓存、排序、回填、兜底逻辑的改动，是否会影响未在 MR 描述中提到的结果集行为
- 配置驱动能力上线后，是否可能让历史配置、旁路配置或其他业务配置出现意外行为

如果发现这类风险，默认表达方式也要克制且具体：

- 明确指出“这不是本次改动的直接目标，但可能受影响”
- 尽量点出受影响的外围业务或调用场景
- 如果只是潜在风险，就写成“建议确认”或“建议补一轮联调验证”，不要伪装成已经发生的 bug

### 3.4 对低价值问题保持克制

默认不把以下内容当成主要输出，除非它们已经影响理解或埋下缺陷：

- 纯格式问题
- 个人风格偏好
- 不影响行为的轻微命名问题
- 可以由格式化工具或 lint 自动处理的问题

还要额外明确一条收敛规则：

- 不对“断言写得是否足够细”“日志打得是否足够精确”这类问题默认上纲上线

当前默认判断是：

- 有日志，通常就比完全无观测更重要，不要求日志必须达到非常细的字段级精确度
- 有断言、有测试覆盖，通常就比追求断言写法是否最严密更重要
- 评审重点是有没有遗漏关键场景、关键分支、关键保护，而不是对已经存在的断言粒度或日志颗粒度较真

只有在下面这些情况下，这类问题才值得被提升为主要问题：

- 关键失败路径完全没有日志，导致线上排查几乎无抓手
- 日志缺失到无法确认核心输入、核心分支或核心失败原因
- 测试虽然存在，但没有覆盖真正容易出错的关键分支
- 断言弱到即使行为已经明显回归，测试仍然可能继续通过

### 3.5 结论必须能指导下一步动作

默认不只报问题，还要给动作建议。建议优先是下面几类：

- 补测试
- 补边界处理
- 做兼容验证
- 增加回滚/兜底方案
- 补日志、监控或告警

这里的“补日志、监控或告警”也按收敛口径理解：

- 重点补关键遗漏
- 不默认要求把已有日志打磨到非常细
- 不把“日志还能再更精准一点”当作阻断合并的理由

### 3.6 单元测试只作为证据，不作为默认发难点

默认对单元测试要明显克制，不把它当成最容易下手的挑刺入口。

当前默认原则是：

- 不因为“覆盖率看起来还能更高”“case 还能更多”“断言还能更细”就直接提出主要问题
- 不把“测试还不够全”当成万能结论
- 只有当测试缺口已经影响对高风险行为的判断时，才值得提出来

更具体地说，只有下面这些情况，才应把单测问题提升为主要问题：

- 当前改动引入了明确的高风险契约或兼容语义，但现有测试完全没有覆盖这类核心行为
- 从实现上已经看到明显容易出错的关键分支，而现有测试会让这种回归轻易漏过去
- 开发把一个通用能力或基础组件抽象出来，但最关键的行为边界没有任何证据锁定

反过来，下面这些默认不该成为主要问题：

- “理论上还能再补几个 case”
- “assert 可以再写细一点”
- “覆盖率不够漂亮”
- “happy path 比异常路径测得多”

除非你能把它直接绑定到一个具体且重要的遗漏风险，否则不要把单元测试本身写成主要问题。

## 4. 结论判定标准

当前默认结论有三档：

### `approve`

适用条件：

- diff 中未发现明确阻断问题
- 存在的剩余风险可接受
- 即使通过，也仍建议人工抽样复核后合并

### `comment`

适用条件：

- 有需要关注的点，但证据不足以直接阻断合并
- 需要补充验证、补测试或补背景说明
- 变更存在一定不确定性，但尚未看到明确错误

### `request_changes`

适用条件：

- 存在较明确的正确性问题
- 存在明显回归风险或兼容性破坏
- 涉及数据、鉴权、安全、核心流程的高风险缺陷
- 改动风险很高，但缺少必要保护、验证或兜底

简单说：

- `approve` = 没看到明确阻断项
- `comment` = 有顾虑，但证据还不够硬
- `request_changes` = 已经看到足够明确的问题或高风险缺口

## 5. 风险等级标准

当前默认风险等级有三档：

### `low`

- 影响面较小
- 即使出问题，后果通常可控
- 变更更偏局部或非核心路径

### `medium`

- 有真实回归可能
- 影响链路比较重要
- 需要补验证或补测试才能放心

### `high`

- 涉及核心交易、登录鉴权、数据写入、外部契约、基础依赖
- 一旦出错，后果明显且回收成本高
- 已看到较高概率的线上风险

风险等级关注的是“出事后的影响面和严重度”，不完全等于 verdict。理论上存在“先不阻断，但风险偏高，需要强提醒”的情况。

## 6. 输出要求

当前默认输出结构应满足：

- 先说明 MR 在做什么
- 再给评审摘要
- 明确给出 verdict
- 明确给出 risk_level
- 列出主要问题
- 列出建议动作

默认表达要求：

- 使用简洁中文
- 问题描述尽量具体
- 建议要可执行
- 不写多余寒暄和铺垫

## 7. 当前实现里已经落地的行为

这部分不是原则，而是当前代码真实在执行的事实。

### 7.1 当前 review 输入

当前 reviewer prompt 实际只喂入：

- MR 标题
- MR 描述
- source -> target 分支
- 作者
- URL
- 每个变更文件的 diff

这意味着当前系统还没有：

- 仓库级上下文检索
- 调用测试结果作为证据
- 语言/框架专项规则引擎
- 基于文件类型的强制检查清单

### 7.2 当前 reviewer 输出契约

当前 reviewer 被要求输出严格 JSON，字段固定为：

- `mr_purpose`
- `summary`
- `verdict`
- `risk_level`
- `findings`
- `suggestions`

其中：

- `verdict` 只能是 `approve` / `comment` / `request_changes`
- `risk_level` 只能是 `low` / `medium` / `high`
- `findings` 和 `suggestions` 是短条目列表

### 7.3 当前评论风格

GitLab 评论现在固定会输出：

- 变更意图
- 审查结论
- 评审摘要
- 主要问题
- 建议动作

并且始终保留一条人工复核说明：自动化审查仅基于 MR diff，合并前仍应结合业务上下文做人审。

### 7.4 当前系统的保守策略

当 AI reviewer 或 OpenClaw reviewer 不可用时，当前系统会降级为基础摘要，而不是中断整个流程。此时默认：

- `verdict = comment`
- `risk_level = medium`
- findings 会写明 fallback 原因

这反映的默认原则是：宁可保守提醒，也不在缺失判断能力时给出乐观结论。

## 8. 目前还没有显式固化，但值得后续加进去的规则位

下面这些是我建议预留给你后面追加特殊要求的位置：

- 哪些问题必须直接 `request_changes`
- 哪些问题只能提醒，不能阻断
- 对测试缺失的容忍度
- 对接口变更、字段变更、兼容性问题的强制检查
- 对安全、权限、SQL、缓存、并发的专项检查清单
- 不同语言或框架的差异化规则
- 评论语气、格式、详细程度
- 对误报率和漏报率的偏好

## 9. 一句话版本

当前默认 CR 原则可以概括为：

基于 MR diff 做保守、可辩护、风险优先的审查，重点抓正确性、回归、兼容性和高影响问题；证据不足时提醒而不武断阻断；输出必须简洁、具体、可执行，并明确保留人工复核责任。

补一条更贴近你刚才要求的表述：

自动化 CR 的重点是发现开发人员漏掉了什么，而不是对断言写法、日志精度、表达颗粒度做吹毛求疵式挑刺。
